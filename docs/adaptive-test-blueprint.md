## 2. Какие специальности должен различать тест

### 2.1. Целевые рекомендации

Тест должен уметь различать 10 направлений:

1. `09.02.01 ТЭ`
2. `09.02.06 ТСА`
3. `09.02.07 ТИП (программист)`
4. `09.02.07 ТБД (администратор баз данных)`
5. `11.02.16 ТМО`
6. `11.02.17 ТУС`
7. `11.02.18 ТСР`
8. `15.02.10 ТМР`
9. `27.02.04 ТУ`
10. `54.02.01 ТД`

### 2.2. Главные смысловые кластеры

Чтобы тест был ветвящимся и устойчивым, сначала он определяет не конкретную специальность, а широкий кластер интересов:

1. `IT`  
   Разработка, программные системы, базы данных, цифровая логика.

2. `NET`  
   Сети, серверы, инфраструктура, комплексные компьютерные системы, телеком.

3. `ELEC`  
   Электроника, пайка, диагностика, платы, микроконтроллеры, устройства.

4. `AUTO`  
   Автоматизация, ПЛК, датчики, робототехника, технологические процессы.

5. `DES`  
   Дизайн, визуальная среда, интерфейсы, предметные и интерьерные решения.

### 2.3. Внутренние специализаторы

После определения широкого кластера тест должен различать более узкие профили:

- `APP` — прикладная разработка, логика, API, программные продукты
- `DB` — данные, SQL, структура, хранение, целостность
- `SYS` — комплексные компьютерные системы: железо + ПО + сеть + защита
- `ADMIN` — сетевое и серверное администрирование
- `TEL` — связь, радиоканалы, мобильная/вещательная инфраструктура
- `REPAIR` — диагностика и ремонт электроники
- `EMB` — создание электронных устройств и встраиваемых систем
- `ROBO` — мехатроника и робототехнические системы
- `PROC` — управление технологическими процессами, КИПиА, АСУ

### 2.4. Поведенческие модификаторы

Они не выбирают специальность сами по себе, но помогают в спорных случаях и нужны для объяснения рекомендации:

- `SEC` — приоритет надёжности, защиты, контроля рисков
- `HANDS` — готовность работать руками с оборудованием
- `IND` — интерес к промышленной / производственной среде
- `PEOPLE` — склонность к координации, согласованию, презентации
- `CREATE` — тяга к созданию нового, а не только к сопровождению

## 3. Как эти шкалы связаны со специальностями

Финальный результат считается в два шага:

1. Сначала по вопросам набираются баллы по кластерам, специализаторам и модификаторам.
2. Затем вычисляются баллы конкретных специальностей.

### 3.1. Формулы специальностей

Ниже рабочие формулы, которые можно прямо переносить в код.

| Специальность | Формула итогового балла |
| --- | --- |
| `09.02.07 ТИП` | `IT * 0.60 + APP * 0.35 + CREATE * 0.05` |
| `09.02.07 ТБД` | `IT * 0.45 + DB * 0.45 + SEC * 0.10` |
| `09.02.01 ТЭ` | `NET * 0.30 + SYS * 0.35 + IT * 0.15 + SEC * 0.10 + HANDS * 0.10` |
| `09.02.06 ТСА` | `NET * 0.50 + ADMIN * 0.30 + SEC * 0.10 + HANDS * 0.10` |
| `11.02.18 ТСР` | `NET * 0.40 + TEL * 0.35 + SEC * 0.10 + HANDS * 0.10 + IND * 0.05` |
| `11.02.16 ТМО` | `ELEC * 0.50 + REPAIR * 0.35 + HANDS * 0.15` |
| `11.02.17 ТУС` | `ELEC * 0.40 + EMB * 0.40 + CREATE * 0.10 + HANDS * 0.10` |
| `15.02.10 ТМР` | `AUTO * 0.45 + ROBO * 0.35 + HANDS * 0.10 + CREATE * 0.05 + IND * 0.05` |
| `27.02.04 ТУ` | `AUTO * 0.45 + PROC * 0.35 + IND * 0.10 + SEC * 0.05 + HANDS * 0.05` |
| `54.02.01 ТД` | `DES * 0.70 + CREATE * 0.20 + PEOPLE * 0.10` |

### 3.2. Как интерпретировать близкие результаты

- если `1 место` опережает `2 место` на `8+` баллов, рекомендация считается уверенной;
- если отрыв `4–7` баллов, показываем основную рекомендацию и “очень близкую альтернативу”;
- если отрыв меньше `4` баллов, в интерфейсе нужно показывать 1 основную и 2 почти равные альтернативы с пояснением, почему они близки.

## 4. Как пользователь получает ровно 10 вопросов

## 4.1. Общая схема

Каждый пользователь проходит:

1. `Q1` — фиксированный
2. `Q2` — фиксированный
3. `Q3` — общий
4. `Q4` — общий
5. `Q5` — общий
6. `Q6/Q7` или другой парный блок по `кластеру №1`
7. `Q8/Q9` или другой парный блок по `кластеру №2`
8. `Q16–Q20` — один контрольный вопрос

Итого:

- `5` общих вопросов для всех
- `4` ветвящихся вопроса по двум лидирующим кластерам
- `1` уточняющий контрольный вопрос

Всего: `10`.

## 4.2. Порядок ветвления

После `Q1–Q5` считаем текущие баллы по кластерам:

- `IT`
- `NET`
- `ELEC`
- `AUTO`
- `DES`

Дальше:

1. Берём `2` кластера с максимальным баллом.
2. По каждому из этих двух кластеров задаём его специализированную пару вопросов.
3. После этого считаем предварительный рейтинг конкретных специальностей.
4. Если внутри лидирующей пары есть неясность, задаём один контрольный вопрос.

## 4.3. Какие пары вопросов принадлежат каким кластерам

| Кластер | Специализированные вопросы |
| --- | --- |
| `IT` | `Q6`, `Q7` |
| `NET` | `Q8`, `Q9` |
| `ELEC` | `Q10`, `Q11` |
| `AUTO` | `Q12`, `Q13` |
| `DES` | `Q14`, `Q15` |

## 4.4. Когда какой контрольный вопрос задаётся

| Вопрос | Когда задаётся |
| --- | --- |
| `Q16` | если спор идёт между `ТИП`, `ТБД`, частично `ТЭ` |
| `Q17` | если спор идёт между `ТЭ`, `ТСА`, `ТСР` |
| `Q18` | если спор идёт между `ТМО` и `ТУС` |
| `Q19` | если спор идёт между `ТМР` и `ТУ` |
| `Q20` | если дизайн конкурирует с техническим направлением или нужно понять, технарь это или дизайнер |

Правило выбора:

1. Сначала смотрим на `топ-2 специальности` после 9-го вопроса.
2. Если они внутри одной семьи, задаём контроль семьи.
3. Если одна из них `54.02.01 ТД`, задаём `Q20`.
4. Если спор между софтом и инфраструктурой, задаём тот контрольный вопрос, который лучше проверяет источник неопределённости:
   - если человек колеблется между `код/данные`, задаём `Q16`;
   - если между `инфраструктура/связь/системная интеграция`, задаём `Q17`.

### Логика влияния Q1

Каждая выделенная единица добавляет баллы по формулам:

| Поле | Влияние |
| --- | --- |
| `code` | `IT +0.06`, `APP +0.04`, `EMB +0.01`, `ROBO +0.01`, `CREATE +0.01` |
| `tests` | `NET +0.03`, `DB +0.02`, `PROC +0.02`, `SEC +0.04` |
| `docs` | `DB +0.02`, `PROC +0.02`, `PEOPLE +0.02` |
| `design` | `DES +0.06`, `CREATE +0.02`, `EMB +0.01` |
| `meeting` | `PEOPLE +0.05`, `CREATE +0.01`, `PROC +0.01` |

Почему это важно:

- высокий `code` сильно тянет в `ТИП`, частично в `ТУС`, `ТМР`, `ТЭ`;
- высокий `tests` поднимает интерес к надёжности, контролю и системному мышлению, что полезно для `ТСА`, `ТЭ`, `ТУ`, `ТБД`;
- высокий `docs` часто коррелирует с аккуратной структурной работой, полезной для `ТБД` и `ТУ`;
- высокий `design` почти всегда сигналит о возможной близости к `ТД`;
- высокий `meeting` не выбирает специальность сам, но усиливает направления, где нужно согласовывать требования и объяснять решения.

### Логика влияния Q2

Позиции в списке дают вес:

- `1 место` = `5` очков
- `2 место` = `4` очка
- `3 место` = `3` очка
- `4 место` = `2` очка
- `5 место` = `1` очко

Далее вес кризиса раскладывается так:

| Кризис | Влияние за каждую единицу веса |
| --- | --- |
| `server` | `NET +1.0`, `ADMIN +0.8`, `SYS +0.6`, `SEC +0.2` |
| `req` | `IT +0.8`, `APP +0.8`, `DES +0.4`, `PEOPLE +0.4` |
| `conflict` | `PEOPLE +1.0`, `PROC +0.3`, `DES +0.2` |
| `vuln` | `SEC +1.0`, `NET +0.7`, `DB +0.4`, `SYS +0.5`, `TEL +0.3` |
| `legacy` | `IT +0.7`, `APP +0.6`, `DB +0.5`, `SYS +0.6` |

Почему это важно:

- если человек ставит `server` очень высоко, это сильный сигнал в сторону `ТСА`, `ТЭ`;
- если первым идёт `vuln`, усиливаются направления с приоритетом надёжности и защиты: `ТЭ`, `ТСА`, `ТБД`, частично `ТСР`;
- если высоко `legacy`, часто это человек, которому интересно разбираться в системной логике и архитектуре;
- если первым ставится `req`, это сдвигает в сторону программной и продуктовой работы;
- если первым идёт `conflict`, это не делает человека “не технарём”, но усиливает роль коммуникации и управления.

## 6. Подробная логика принятия решения

## 6.1. Шаг 1. Получаем стартовый профиль по Q1–Q5

После первых пяти вопросов:

- становится ясно, в каких двух кластерах у пользователя максимум интереса;
- видно, цифровой он, аппаратный, производственный или визуальный;
- видна форма интереса: создавать, обслуживать, чинить, координировать, защищать.

На этом этапе система ещё не должна жёстко рекомендовать специальность.  
Её задача — сузить пространство.

## 6.2. Шаг 2. Углубляемся только в два лидирующих кластера

Это главный принцип адаптивности.

Пример:

- если после `Q1–Q5` лидируют `NET` и `AUTO`, пользователь получает `Q8`, `Q9`, `Q12`, `Q13`;
- если лидируют `IT` и `DES`, он получает `Q6`, `Q7`, `Q14`, `Q15`;
- если лидируют `ELEC` и `AUTO`, он получает `Q10`, `Q11`, `Q12`, `Q13`.

Таким образом пользователь не тратит время на нерелевантные вопросы.

## 6.3. Шаг 3. Выявляем, к чему он ближе внутри родственных направлений

Основные развилки внутри банка:

- `ТИП` vs `ТБД`
- `ТЭ` vs `ТСА` vs `ТСР`
- `ТМО` vs `ТУС`
- `ТМР` vs `ТУ`
- `ТД` vs “технический человек с креативной жилкой”

Именно для этого нужны вопросы `Q16–Q20`.

## 6.4. Шаг 4. Считаем финальный рейтинг специальностей

После 10-го вопроса:

1. Считаются все промежуточные шкалы.
2. По формулам из раздела 3 вычисляются финальные баллы специальностей.
3. Результаты сортируются по убыванию.

Система должна возвращать:

- `1 основную рекомендацию`
- `2 альтернативы`
- `короткое объяснение`, за счёт каких шкал человек туда попал

## 7. Что именно должно объяснять пользователю результат

Рекомендация должна звучать не так:

`Тебе подходит 09.02.06.`

А так:

`Тебе ближе направления, где нужно поддерживать сложную техническую среду, быстро реагировать на сбои и держать под контролем сеть, доступы и надёжность. Поэтому у тебя высоко вышла 09.02.06 ТСА. Близкие альтернативы: 09.02.01 ТЭ и 11.02.18 ТСР.`

### 7.1. Шаблон объяснения

Объяснение лучше строить из трёх частей:

1. `Что тебе нравится делать`
   Например: писать логику, искать сбои, чинить устройства, настраивать процессы, создавать визуальные решения.

2. `В какой среде тебе комфортнее`
   Например: кодовая база, сеть и серверы, лаборатория электроники, цех/линия, дизайн-студия.

3. `Почему именно эта специальность`
   Например: потому что в ней сходятся сеть + защита + работа с инфраструктурой.

### 7.2. Примеры объяснений по направлениям

`09.02.07 ТИП`  
Подходит тем, кто хочет создавать цифровые продукты, писать логику и собирать функциональность в работающий сервис.

`09.02.07 ТБД`  
Подходит тем, кому ближе структура, данные, SQL, целостность, хранение и надёжность информации.

`09.02.01 ТЭ`  
Подходит тем, кто мыслит компьютерную среду целиком: техника, сеть, программное обеспечение, безопасность.

`09.02.06 ТСА`  
Подходит тем, кому интереснее эксплуатация сети, серверов, доступов, администрирование и стабильная работа инфраструктуры.

`11.02.18 ТСР`  
Подходит тем, кому интересны каналы связи, сигнал, выездная настройка и техническая связь “в поле”.

`11.02.16 ТМО`  
Подходит тем, кто любит диагностику, ремонт, схемы, приборы и возвращение техники к жизни.

`11.02.17 ТУС`  
Подходит тем, кто хочет не только чинить, но и создавать электронные устройства, платы и встраиваемые системы.

`15.02.10 ТМР`

## 8. Важные методические принципы

### 8.1. Чего здесь нельзя делать

- нельзя задавать вопросы в лоб вроде `Хочешь быть программистом?`;
- нельзя строить результат только на одном вопросе;
- нельзя интерпретировать высокий креатив как автоматический выбор дизайна;
- нельзя интерпретировать любовь к технике как автоматический выбор электроники: это может быть и `ТЭ`, и `ТСА`, и `ТМР`, и `ТУ`.

### 8.2. Что здесь важно сохранить

- первые два вопроса уже хорошие, потому что они измеряют не декларируемый интерес, а стиль мышления и поведения;
- тест должен мерить не только `что нравится`, но и `в какой среде человек готов этим заниматься`;
- рекомендации должны быть объяснимыми преподавателю, абитуриенту и родителю.

## 10. Итоговое решение

Предлагаемая модель хороша тем, что:

- укладывается ровно в `10` вопросов на пользователя;
- использует `20` вопросов в полном банке;
- сохраняет уже сделанные сильные `Q1` и `Q2`;
- учитывает все специальности из исходного документа, а не только цифровой блок;
- даёт не просто итог, а понятную и защищаемую логику, почему рекомендовано именно это направление.

Следующий практический шаг после этого документа:

1. перевести этот blueprint в `json`-схему вопросов;
2. реализовать движок скоринга;
3. подключить ветвление по кластерам прямо в интерфейсе теста.
