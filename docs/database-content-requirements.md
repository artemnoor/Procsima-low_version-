# Требования к содержимому БД

## Цель документа

Этот документ описывает, какие данные должны храниться в базе данных для сайта выбора образовательных направлений. Он составлен с учетом текущего наполнения страниц:

- `index.html` — главная страница платформы
- `catalog.html` — каталог направлений
- `compare.html` — сравнение направлений
- `old.html` — прототип детальной страницы направления
- `test.html` и `test-q1.html` ... `test-q9.html` — профтест
- `dataexample.json` — текущий mock-источник данных по направлениям

Документ нужен для того, чтобы:

- убрать дублирование данных между страницами;
- сделать единый источник правды для каталога, сравнения и детальной страницы;
- подготовить данные для профтеста и страницы результатов;
- заложить структуру под дальнейшее наполнение контентом.

## Главный принцип

В БД должны храниться не только "направления", но и связанные с ними:

- приемные показатели;
- учебные блоки и предметы;
- расширенный контент детальной страницы;
- карьерные траектории;
- данные для профтеста и рекомендаций;
- при необходимости контент главной страницы и FAQ.

Стили, анимации, HTML-разметку и декоративные элементы хранить в БД не нужно.

## Основные сущности

### 1. Направления подготовки

Основная сущность, от которой зависят каталог, сравнение, рекомендации и детальная страница.

Таблица: `programs`

Обязательные поля:

- `id`
- `code` — код направления, например `09.02.01`
- `slug` — человекочитаемый URL, например `kompyuternye-sistemy-i-kompleksy`
- `title` — название направления
- `qualification` — квалификация
- `department_id` — кафедра
- `education_level` — СПО / бакалавриат / специалитет
- `study_form` — очная / заочная / очно-заочная
- `duration` — срок обучения
- `short_description` — короткое описание для каталога и карточек
- `hero_description` — расширенное описание для шапки детальной страницы
- `is_published`
- `sort_order`
- `hero_image_id` — ссылка на основное изображение, если оно будет разным у направлений
- `created_at`
- `updated_at`

Дополнительно можно хранить:

- `seo_title`
- `seo_description`
- `seo_keywords`
- `status` — draft / published / archived

### 2. Кафедры

Нужны как отдельная сущность, чтобы не дублировать одно и то же название кафедры в разных программах и предметах.

Таблица: `departments`

Поля:

- `id`
- `name`
- `short_name`
- `slug`
- `description`
- `contact_email`
- `contact_phone`
- `website_url`
- `is_published`

### 3. Приемные показатели по годам

Нужны для сравнения направлений и построения графиков проходного балла.

Таблица: `program_admission_stats`

Поля:

- `id`
- `program_id`
- `year`
- `budget_places`
- `paid_places`
- `tuition_per_year`
- `passing_score_budget`
- `passing_score_paid`
- `comment` — если нужно пояснение по конкретному году

Почему это отдельно:

- количество мест и проходной балл меняются по годам;
- стоимость обучения тоже может меняться;
- это позволит строить графики без хранения массивов JSON внутри одной записи программы.

### 4. Учебные блоки

Нужны для сравнения программ по укрупненным категориям.

Таблица: `subject_blocks`

Поля:

- `id`
- `name` — например `Программирование`, `Математика`, `Сети`
- `slug`
- `description`
- `sort_order`
- `color` — если блоки будут цветово кодироваться в интерфейсе

### 5. Предметы

Справочник дисциплин, чтобы один и тот же предмет не дублировался строкой в каждой программе.

Таблица: `subjects`

Поля:

- `id`
- `name`
- `slug`
- `description`
- `default_department_id` — опционально
- `default_info_url` — опционально

### 6. Связь направления и предметов

Главная таблица для учебного плана, матрицы блоков и сравнения.

Таблица: `program_subjects`

Поля:

- `id`
- `program_id`
- `subject_id`
- `block_id`
- `teaching_department_id`
- `hours`
- `semester` — опционально
- `course_year` — опционально
- `is_required`
- `info_url`
- `sort_order`

Эта таблица должна обеспечивать:

- список предметов на детальной странице;
- группировку предметов по блокам;
- подсчет часов по блокам;
- матрицу сравнения направлений.

### 7. Документы программы

Нужны для ссылок на учебный план, описание ОПОП и другие документы.

Таблица: `program_documents`

Поля:

- `id`
- `program_id`
- `type` — curriculum / opop / regulation / brochure / other
- `title`
- `file_id` или `url`
- `description`
- `version_label`
- `published_at`
- `is_primary`

## Контент детальной страницы направления

Текущий прототип детальной страницы уже показывает, что одной таблицы `programs` недостаточно. Для направления нужны содержательные блоки, которые можно редактировать отдельно.

### 8. Секции детальной страницы

Таблица: `program_sections`

Поля:

- `id`
- `program_id`
- `section_key`
- `title`
- `body`
- `sort_order`
- `is_published`

Примеры `section_key`:

- `who_is_it`
- `what_you_learn_intro`
- `skills_intro`
- `career_intro`
- `advantages`
- `practice`
- `employment`

### 9. Элементы внутри секций

Нужны для блоков вида "Чему учат?" или "Какие навыки получает выпускник?".

Таблица: `program_section_items`

Поля:

- `id`
- `section_id`
- `title`
- `description`
- `icon`
- `sort_order`

Эта модель подходит для контента:

- "Чему учат?"
- "Какие навыки получает выпускник?"
- "Преимущества программы"
- "Что будет изучаться на практике"

### 10. Карьерные роли

Нужны для блока "Где можно работать?" и для будущих рекомендаций по профессиям.

Таблица: `career_roles`

Поля:

- `id`
- `title`
- `slug`
- `description`
- `is_published`

Связующая таблица: `program_career_roles`

Поля:

- `id`
- `program_id`
- `career_role_id`
- `sort_order`
- `comment`

## Как разложить ваш новый контент по БД

Для направления "Компьютерные системы и комплексы" ваш новый контент должен храниться так:

### Блок "Кто это?"

Формат хранения:

- одна запись в `program_sections`
- `section_key = who_is_it`
- `title = Кто это?`
- `body` = полный текст описания специалиста

### Блок "Чему учат?"

Формат хранения:

- одна запись в `program_sections`
- `section_key = what_you_learn_intro`
- `title = Чему учат?`
- `body` = при необходимости короткое вводное описание

Пункты блока хранятся в `program_section_items`:

- `Аппаратная часть`
- `Сетевые технологии`
- `Программирование`
- `Управление данными`
- `Безопасность`

У каждого пункта:

- `title`
- `description`
- `sort_order`

### Блок "Какие навыки получает выпускник?"

Формат хранения:

- одна запись в `program_sections`
- `section_key = skills_intro`
- `title = Какие навыки получает выпускник?`

Элементы в `program_section_items`:

- `Проектирование`
- `Настройка и монтаж`
- `Диагностика и обслуживание`
- `Оптимизация`
- `Обеспечение безопасности`
- `Сетевое администрирование`
- `Работа с документацией`

### Блок "Где можно работать?"

Формат хранения:

- одна запись в `program_sections`
- `section_key = career_intro`
- `title = Где можно работать?`

Карьерные роли через `career_roles` и `program_career_roles`:

- `Системный администратор`
- `Сетевой инженер`
- `Разработчик программного обеспечения`
- `Администратор баз данных`
- `Специалист по информационной безопасности`
- `Специалист технической поддержки`
- `Аналитик данных`

## Данные для каталога

Чтобы `catalog.html` строился полностью из БД, нужно брать из `programs` и `program_admission_stats`:

- код направления;
- название;
- кафедру;
- короткое описание;
- количество мест;
- ссылку на страницу направления;
- статус публикации;
- изображение или визуальный акцент, если появится.

Для каталога не нужно хранить отдельную таблицу карточек, если все карточки — это просто представление программ.

## Данные для сравнения

Для `compare.html` в БД должны быть доступны:

- основные данные о программе;
- статистика приема по годам;
- стоимость;
- срок обучения;
- список предметов;
- блоки предметов;
- часы по каждому предмету;
- агрегируемые часы по блокам.

Сравнение должно строиться из:

- `programs`
- `program_admission_stats`
- `program_subjects`
- `subjects`
- `subject_blocks`

## Данные для профтеста

Сейчас профтест живет в `localStorage`, но для полноценной работы сайта лучше хранить его структуру в БД.

### 11. Вопросы профтеста

Таблица: `quiz_questions`

Поля:

- `id`
- `key` — например `q1`, `q2`
- `title`
- `description`
- `question_type`
- `sort_order`
- `is_active`

Типы вопросов могут быть такими:

- `slider_distribution`
- `ranking`
- `scenario`
- `bipolar_scale`
- `card_choice`
- `matching`

### 12. Варианты ответов

Таблица: `quiz_question_options`

Поля:

- `id`
- `question_id`
- `key`
- `label`
- `description`
- `icon`
- `color`
- `sort_order`

### 13. Оси интересов и профиля

Таблица: `quiz_traits`

Поля:

- `id`
- `key`
- `title`
- `description`

Примеры осей:

- `code`
- `systems`
- `hardware`
- `analytics`
- `security`
- `teamwork`
- `design`
- `operations`

### 14. Веса ответов

Таблица: `quiz_option_trait_weights`

Поля:

- `id`
- `question_option_id`
- `trait_id`
- `weight`

Для сложных вопросов, где ответ не один, а, например, распределение или ранжирование, можно хранить отдельные правила подсчета в коде, но оси и их связь с направлениями все равно должны быть описаны в БД.

### 15. Профиль направления для рекомендаций

Таблица: `program_trait_scores`

Поля:

- `id`
- `program_id`
- `trait_id`
- `score`

Это позволит:

- считать релевантность направления по результатам профтеста;
- выводить "похожие направления";
- строить страницу результатов теста;
- объяснять пользователю, почему ему рекомендовано именно это направление.

### 16. Прохождения теста

Если результаты теста нужно сохранять:

Таблица: `quiz_attempts`

Поля:

- `id`
- `session_id`
- `user_id` — если будет авторизация
- `started_at`
- `completed_at`
- `result_program_id` — основной рекомендованный вариант
- `result_payload` — итоговый JSON с расшифровкой, если нужно

Таблица: `quiz_attempt_answers`

Поля:

- `id`
- `attempt_id`
- `question_id`
- `answer_payload`

## Файлы и медиа

Если на сайте появятся отдельные изображения программ, PDF учебных планов, иконки, баннеры:

Таблица: `files`

Поля:

- `id`
- `storage_path`
- `original_name`
- `mime_type`
- `size_bytes`
- `alt_text`
- `created_at`

В БД стоит хранить только ссылки и метаданные файлов, а не сами бинарные данные.

## Контент главной страницы

Для текущего этапа это необязательно, но если главную страницу нужно редактировать из админки, стоит предусмотреть:

- `pages`
- `page_sections`
- `faq_items`
- `feature_cards`
- `cta_blocks`

Это нужно для таких блоков, как:

- "Ключевые возможности"
- "Как платформа помогает"
- FAQ
- финальный CTA

Если главная редко меняется, эти блоки можно временно оставить в коде, а в БД хранить только данные по направлениям, сравнению и профтесту.

## Что должно быть обязательным минимумом для первого запуска

Если делать минимально жизнеспособную БД, достаточно следующих таблиц:

- `programs`
- `departments`
- `program_admission_stats`
- `subject_blocks`
- `subjects`
- `program_subjects`
- `program_documents`
- `program_sections`
- `program_section_items`
- `career_roles`
- `program_career_roles`
- `quiz_questions`
- `quiz_question_options`
- `quiz_traits`
- `program_trait_scores`

Этого уже хватит, чтобы:

- собрать каталог из БД;
- собрать детальную страницу из БД;
- построить сравнение;
- подготовить страницу результатов профтеста;
- без переделки добавить ваш новый контент по направлениям.

## Что не стоит хранить в БД

Не нужно хранить в БД:

- CSS;
- SVG-анимации;
- готовый HTML страниц;
- чисто декоративные подписи, если они не редактируются;
- вычисляемые агрегаты, которые легко считаются на лету, если нет проблем с производительностью.

## Ключевой вывод

База данных должна быть построена вокруг сущности направления подготовки, но с обязательным разделением на:

- справочные данные;
- приемную статистику по годам;
- учебный контент;
- контент детальной страницы;
- карьерные выходы;
- данные профтеста и рекомендательной логики.

Именно такая структура позволит перестать поддерживать отдельно `catalog.html`, `compare.html`, `old.html` и JSON-моки, а вместо этого собрать весь сайт из одного набора согласованных данных.
